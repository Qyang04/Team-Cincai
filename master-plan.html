Product Core

Not a claims app. A finance operations control plane.

Every case must become understandable, controllable, reconcilable, and postable. That is the governing idea behind the full system.

Persistent case object

All artifacts, extractions, questions, policy outcomes, approvals, exports, and audit events attach to one stateful case.

Governed AI

GLM remains central for extraction and reasoning, but output is bounded by schemas, confidence, provenance, and human-controlled gates.

Accounting-ready completion

Completion means the normalized financial payload is valid and exportable, not merely that the UI looks complete.

Platform Modules

The master plan names the core modules explicitly.

These modules make the plan easier to map to team ownership, implementation boundaries, and demo narrative.

Claim & Case Management

Owns case lifecycle, artifact viewer, timeline, transitions, assignment, and case detail surfaces.

AI Extraction & Reasoning

Parses messy inputs, extracts structured fields, generates clarification prompts, and supports FX estimation and reconciliation context.

Fraud & Duplicate Detection

Uses extract and reason stages to detect exact duplicates, near duplicates, suspicious resubmissions, and anomaly signals before approval or export.

Policy Engine & Configuration

Owns routing rules, spend thresholds, required fields, reconciliation checks, and admin policy configuration.

User Management & RBAC

Owns authentication, organizational mapping, routing identity, and authorization boundaries.

Audit, Reporting & Export

Owns immutable logging, finalized accounting payload generation, CSV or API handoff, export tracking, and basic operational analytics.

State Machine

Official lifecycle states.

The runtime uses one canonical MVP state machine across all workflows. Deeper approval, FX, and tax logic operate inside these gates instead of creating extra top-level states.

    AWAITING_REQUESTER_INFO --> POLICY_REVIEW: requester answered

    AWAITING_APPROVAL --> APPROVED
    AWAITING_APPROVAL --> REJECTED

    FINANCE_REVIEW --> APPROVED
    FINANCE_REVIEW --> REJECTED
    FINANCE_REVIEW --> AWAITING_REQUESTER_INFO: send back
    FINANCE_REVIEW --> EXPORT_READY
    FINANCE_REVIEW --> RECOVERABLE_EXCEPTION

    APPROVED --> EXPORT_READY
    EXPORT_READY --> EXPORTING
    EXPORTING --> EXPORTED
    EXPORTING --> RECOVERABLE_EXCEPTION: export failure

    RECOVERABLE_EXCEPTION --> INTAKE_PROCESSING: retry intake
    RECOVERABLE_EXCEPTION --> POLICY_REVIEW: retry policy
    RECOVERABLE_EXCEPTION --> EXPORT_READY: retry export

    REJECTED --> CLOSED

Deep Finance Controls

Real finance nuance is in scope at the architecture level now.

The prototype stays focused, but the contracts and system model must already support the real-world finance controls that matter.

Approval matrix

Support sequential, parallel, conditional, and delegated approvals rather than assuming one flat approver forever.

FX reconciliation

Distinguish estimated market-rate conversion from realized settled amount backed by card or bank evidence.

Tax segregation

Normalize net, tax, and gross amounts and validate arithmetic before export readiness.

Fraud & Duplicate Detection: EXTRACT

  • receipt or invoice number
  • merchant, vendor tax ID, and OCR text
  • amount, currency, spend date, and project code
  • filename, uploader identity, and artifact hash where available

Fraud & Duplicate Detection: REASON

  • classify exact duplicate versus near duplicate
  • detect suspicious resubmission or conflicting submission
  • generate rationale summary for reviewers
  • route to warning, finance review, or recoverable exception

Repository status

What is already built versus what is still open.

This is the execution-facing view: in the repo today versus not done yet, aligned with master-plan-repo-status-and-execution-guide.md (inspection snapshot 2026-04-23). The rest of this HTML page describes the target architecture; this section tracks delivery reality.

The codebase has a strong workflow backbone (states, transitions, case detail, approvals, finance review, export, recoverable exceptions, mock-first seams). The largest gap is finance depth: multi-tier approvals, FX and tax behavior, richer duplicate signals, evidence-driven extraction, and production-grade auth and connectors.

Case lifecycle and orchestration

In the repo today

  • Canonical workflow enum and transition rules in shared contracts; enforcement in the API.
  • Orchestrated submission from draft through intake processing.
  • Policy routing job, export job, recoverable exception re-entry.
  • End-to-end path through clarification or policy, approval or finance review, export or recoverable exception.

Not done yet

  • Explicit stage registry (entry or exit conditions, success or failure transitions, stage-level audit rules).
  • First-class manual-owner metadata at human gates.
  • Richer failure categories and stage-specific retry metadata.

AI intake and extraction

In the repo today

  • Mock extraction shape, mock clarification, mock routing wired through the flow.
  • Optional Z.AI provider path alongside mock AI.

Not done yet

  • Evidence-aware extraction from uploads (OCR, real text, evidence-linked metadata).
  • Broader fields and stronger per-field confidence and provenance.
  • Invalid or unsafe AI output handling beyond basic failure paths.
  • Workflow-specific prompting and extraction rules.

Policy and routing

In the repo today

  • Project code requirement by workflow; amount threshold routing; vendor invoice number warning.
  • Simple duplicate signals (for example filename); admin thresholds and basic routing defaults.

Not done yet

  • Reconciliation flags and approval-requirement outputs driving export readiness in depth.
  • Workflow-specific policy packs; richer duplicate and anomaly reasoning with cross-case signals.
  • Tax and arithmetic checks; FX policy (estimate vs proof vs escalation).

Approvals

In the repo today

  • Approval queue; approve, reject, request-info; reopen after requester answers.
  • Routing-configured default approver; single-step approval for the MVP path.

Not done yet

  • Approval matrix persistence and multi-task patterns.
  • Sequential, parallel, and conditional tiers; delegation and substitutes.
  • Matrix-level UI (pending vs completed, stage progress).

Finance review

In the repo today

  • Finance queue; approve, reject, send-back; policy routing into finance.
  • Send-back into clarification and resume.

Not done yet

  • Richer reason categories and finance coding or resolution fields.
  • Reconciliation-specific resolution data (settled amount, evidence notes).
  • Queue assignment and case ownership in the finance queue.

Export, audit, admin, auth, cross-cutting

In the repo today

  • Export or audit: export-ready, processing, recoverable export failure; audit and transitions in UI.
  • Admin: policy settings, routing settings, connector status surface.
  • Auth: request user resolution, role guards, mock headers, optional Supabase JWT path.
  • Shared or schema or web or tests: core enums and models, case detail contracts, main web surfaces, API and schema tests, build green, mock integrations and BullMQ-ready queue.

Not done yet

  • Export: normalized accounting payload spec, handoff history, retry metadata, operational analytics.
  • Admin: matrix config, workflow-specific policy editing, FX or tax policy, connector credentials and change history.
  • Auth: real sign-in and session UX, org and manager mapping, approver identity beyond defaults.
  • Contracts or DB: stage and matrix contracts, reconciliation and anomaly persistence, assignment fields.
  • Web or integrations or tests: inbox filtering and assignment, matrix or finance UI depth, real OCR or notifications or export or directory, web and integration tests and seeded demo verification.

Implementation Sequence

Build depth where the prototype wins, then broaden safely.

The roadmap balances team execution, demo clarity, and architecture integrity.

Phase 1

Stabilize the current prototype: fix build issues, align API and web behavior, confirm documented state transitions.

Phase 2

Formalize Harness runtime metadata: stage, manual action required, reasoning summary, failure mode, and better view models.

Phase 3

Deepen finance correctness: approval matrix contracts, FX reconciliation fields, tax normalization, and recoverable exception categories.

Phase 4

Polish the flagship expense-claim demo while keeping the shared engine broad enough for the other workflow types.

Decision Summary

The master spec locks one strong direction.

This project should be built as a case-first finance control plane with governed AI, explicit workflow stages, richer finance contracts, and visible exception recovery.

Keep

  • Modular monolith architecture
  • Mock-first adapters
  • Case-first workflow model
  • Shared contracts in the workspace package
  • Human approval as mandatory control

Add next

  • Canonical runtime stages
  • Approval matrix support
  • FX reconciliation fields
  • Tax-aware normalized exports
  • Explicit recoverable exception semantics
